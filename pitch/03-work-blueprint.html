The same operating model on Azure DevOps + GitHub, at team scale

Document 2 describes a pipeline proven in production with one human, one repo, and GitHub issues. This page re-bases that exact operating model onto our stack — Azure DevOps work items, quarter/sprint cadence, our six work-item states, our grill-* skills, and our generic TDD agents — and extends it across many teams and repositories. Nothing here is speculative: every mechanism below is either a translation of the running personal pipeline or a skill already in daily use at work, each translation is called out explicitly, and the handful of deliberate deltas are listed in section 7.

1 · Rosetta stone: personal project → enterprise

2 · Work-item hierarchy and planning cadence

Planning happens at three altitudes with a grill skill guarding each descent. A Feature cannot enter a quarter until grill-azdo-feature has forced it into a consistent, decomposable state; a Story cannot reach Ready until grill-azdo-story has done the same at story grain. This is the direct enterprise analog of Doggiehood's "no work is eligible until analysis produced an approved Build checklist" — the grill skills are the analysis stage, made interactive.

3 · The story lifecycle: our six states as the gating sections

This is Document 2's diagram, re-drawn with our states. The mapping is almost embarrassingly clean: New is intake, Research is the AI analysis section (with the same two "awaiting a human" sub-states, carried as tags), Ready is the approved dev queue, Develop is the overnight build, Done is merged-and-verified, and Closed is product-owner acceptance. The two human loops (/revise, /redo) and the parking side-rail translate unchanged.

Research runs in two complementary modes. When the team lead or PO has time to engage, grill-azdo-story runs as a live session that interrogates them until the story can't be misread. Between sessions the story simply waits in Research under one of the two tags below, and comment-based /revise notes are the asynchronous fallback — feedback left overnight is picked up by the next Research pass, exactly as in the personal project. Authority is one consistent rule throughout: the team lead approves stories, the PO approves features, and either may admit.

One deliberate strengthening versus today's manual process: Ready stops being an opinion. A story reaches Ready only through the grill exit + explicit /approve, so "Ready" always means the same thing — testable acceptance criteria, routed repo, resolved dependencies, assigned sprint. That consistency is precisely what makes overnight agent development safe.

4 · The feature lifecycle: quarter-gated, then delegated

Features run the same shape one level up and one cadence slower. The critical move is the hand-off: when a Feature reaches Ready, planning authority transfers from the portfolio level to the owning team — the Feature's output is a set of linked child Stories in that team's backlog, each of which then runs the story lifecycle above under that team's own gatekeepers. This is how one pipeline design serves many teams without central bottlenecks.

5 · Which skills and agents run at which phase

Same six-column shape as the personal project, with our skills slotted in. The grill family replaces the fire-and-forget analysis skill — a deliberate upgrade for enterprise ambiguity: instead of one AI pass that either plans or asks a single question, the grill skills interrogate the human until the work item can't be misread, then write the result back. The deterministic layer (parser, queue selector, dashboard renderer) is ported nearly as-is; it is stack-agnostic Python against a different REST API, and stays unit-tested in CI.

6 · Scale-out: many teams, many repos, one operating model

The personal project is a single instance of a pattern that federates naturally, because every coupling point is already explicit: stories carry their repo, dependencies are typed links, and each gate names its authorized role. The org runs many small pipelines that share one design — not one big pipeline. A platform team owns the shared skill/agent definitions and the deterministic scripts; each product team owns its own gates, backlog, and caps; each repo gets its own nightly builder and PR stream.

7 · What genuinely changes from the personal project

  • Authority becomes role-based. "Repo owner only" becomes a small allowlist per gate (PO at feature gates, team lead at story gates), still enforced in the deterministic parser — config per team, not per person.
  • Two commands are re-based onto ADO's planning fields. /focus disappears — sprint assignment already expresses "build this now", so the queue selector filters on the active iteration instead of a milestone marker. /milestone becomes /sprint <iteration> on stories and /quarter <Q> on features — same role as before: an explicit override of the AI-proposed placement, resolved at /approve time by the same parser rule.
  • Analysis becomes conversational. The grill skills interrogate a human to reach Ready, where the personal project's analysis runs unattended and hands back one question at a time. Both end in the same artifact: approved, TDD-ordered acceptance criteria.
  • Two systems instead of one. Work state lives in ADO, code and CI in GitHub. The joint is the story's repo-routing field plus PR ↔ story links written by the builder; the dashboard renders both sides.
  • Watermark mechanics. GitHub comment reactions become ADO comment likes from the bot identity (or a processed-comment-ID ledger on the work item if likes prove unreliable) — same idempotency contract either way.
  • Caps become budget policy. One repo × 3 issues/night becomes a per-repo, per-team cap set in pipeline config — the org's model-spend dial, adjustable without touching any logic.

8 · The questions engineers ask first

What stops an agent from merging bad code?

Structure, not trust: agents have no merge rights. They open PRs, keep CI green, and stop. A human reviews every diff against acceptance criteria that a human already approved. Branch protection makes this a platform guarantee rather than a prompt instruction.

What stops a random commenter from driving the pipeline?

The command parser honors only allowlisted identities per gate, and every honored or ignored command is reported in the routine's summary. This rule lives in unit-tested Python, not in the model — the ported parse_commands.py keeps its test suite.

What happens when the AI's plan is wrong?

The same thing that happens today when a human's plan is wrong, but cheaper: /revise with notes sends it back through Research with your feedback attached; /redo starts the analysis clean. Nothing builds until you approve, so a wrong plan costs a comment, not a sprint.

What happens when a story fails to build overnight?

Its commits are dropped so the shared branch stays green, the story returns to Ready with the failure reason recorded, and the run's PR lists what was dropped and what was capped out. Silent partial success is designed out: the report always distinguishes built / dropped / deferred.

How are dependencies between teams handled?

As ADO predecessor–successor links, created during Research. Every repo's queue selector treats an unmet predecessor as a hard block, so a story simply never becomes eligible until the other team's story is Done — coordination by data, not by meetings.

Where does this run, and with what credentials?

Scheduled agent routines (the same mechanism the personal project uses daily) plus cron CI jobs for the deterministic renderers. ADO access via a scoped service identity; GitHub via the existing app installation. Skills, agents, and scripts are version-controlled in a pipeline-config repo, so the pipeline itself ships by PR.

What does it cost, and how is spend controlled?

Model spend concentrates in two places — Research (grilling) and Develop (TDD build) — and both are explicitly gated: nothing enters Research without an admit, and nightly caps bound Develop per repo. Parsing, queueing, dashboards, CI, and releases are deterministic and cost nearly nothing. The caps are the throttle: start at 2–3 stories per repo per night and tune with real data.

How do we know the process is being followed?

The state machine is the process, and scripts move it — a story physically can't reach Develop without approved acceptance criteria, and can't reach Done without a merged PR. The daily dashboard makes queue health and "your move" items visible per role; deviations are self-reported in every PR under a mandatory heading.
