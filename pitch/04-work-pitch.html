The Agentic Delivery Pipeline
Humans decide. AI executes. Scripts enforce.

An operating model for AI-assisted delivery, running in production today: work items flow through our standard Azure DevOps states, every consequential decision passes a named human role, and everything between those decisions — analysis, planning, test-driven implementation, reporting — runs autonomously on a fixed daily schedule. This document is the case for adopting it beyond the teams already using it.

Human decision (role-gated) AI work (skills & agents) Deterministic script (unit-tested) ADO state / tag Planning structure

1 · The problem this solves

Three kinds of waste dominate delivery time, and none of them are coding. Translation waste: turning an idea into work a developer can actually start — requirements churn, follow-up questions, re-litigated scope. Readiness drift: "Ready" means something different on every team and every story, so sprints absorb work that isn't actually buildable. Idle capacity: decisions queue behind calendars, and nothing at all happens overnight. The pipeline attacks all three: it makes readiness a machine-checked fact, compresses every decision to a short comment, and moves the labor of building to hours when no one is waiting on it.

2 · The story lifecycle: our six states as gates

The board's states are the pipeline — no parallel tracking system. A User Story physically cannot reach Develop without approved acceptance criteria, and cannot reach Done without a human-merged pull request. Authorized humans move work forward by typing short /commands in work-item comments; a gatekeeper translates those commands into state moves, honors only allowlisted roles, and marks processed comments so re-runs never double-apply.

4 · Planning altitude: a grill gate at every descent

Work descends from strategy to buildable units through our normal hierarchy, on our normal cadence — Features commit to quarters, Stories to sprints. What's new is that each descent is guarded by a grill skill that interrogates the responsible human until the work item can't be misread, then writes the result back. Ready stops being an opinion: every story that reaches Ready has testable, TDD-ordered acceptance criteria, a routed repo, linked dependencies, and an assigned sprint — every time, on every team.

5 · Which skills run at which phase

Everything AI-facing is a version-controlled skill or agent definition, owned by the platform team and shipped by pull request like any other code. 8 · Why it's safe to run while you sleep

  • Authorization — only allowlisted roles move work; every honored and every ignored command is reported.
  • Idempotency — processed comments are watermarked; re-running a routine with nothing new changes nothing.
  • No invented requirements — Research must stop and ask on any un-specced decision; drafting one requires an explicit /propose.
  • No self-merge — agents open PRs and keep CI green; branch protection means only a human lands code.
  • Honest reporting — every agent PR opens with a mandatory Deviations and Decisions section; nightly caps and dropped stories are logged, so a truncated run never reads as complete.
  • Tested guardrails — the parser, queue selector, and dashboard renderer are plain scripts with unit tests run in CI on every change.

9 · Scale: many teams, many repos, one operating model

This federates naturally because every coupling point is explicit data: stories carry their repo, dependencies are typed links, and each gate names its authorized role. The org runs many small pipelines that share one design. The platform team owns the skill library and scripts; each team owns its gates, backlog, and caps; each repo opts in to a nightly builder. Cross-team work needs no meetings to coordinate: predecessor links block a story's eligibility until the other team's story is Done — the queue selector enforces it every night.

11 · What onboarding a team actually takes

Joining is configuration, not construction — the skills, agents, and scripts already exist in the platform library. A team onboards with one config PR declaring three things:

  • The allowlist — who holds the PO and team-lead roles at each gate.
  • The caps — how many stories per repo per night (start at 2–3; it's the spend dial).
  • The repos — which repositories opt in to a nightly builder. Everything else — boards, CI, branch protection, release tooling — stays exactly as the team runs it today.

12 · Questions engineers ask first

What stops an agent from merging bad code?

Permissions, not promises: agents have no merge rights. They open PRs, keep CI green, and stop. A human reviews every diff against acceptance criteria a human already approved.

What happens when the AI's plan is wrong?

/revise with notes sends it back through Research with your feedback attached; /redo starts clean. Nothing builds until you approve, so a wrong plan costs a comment.

What happens when a story fails to build overnight?

Its commits are dropped so the shared branch stays green, the story returns to Ready with the failure reason recorded, and the PR lists what was built, dropped, and deferred — partial success is never silent.

How are cross-team dependencies handled?

As predecessor–successor links created during Research. A story is simply never eligible to build until its predecessors are Done — coordination by data, not by meetings.

What does it cost, and how is spend controlled?

Model time concentrates in Research and Develop, and both are gated: nothing enters Research without an admit, and nightly caps bound Develop per repo. Parsing, queues, dashboards, and CI are deterministic and near-free. The caps are the throttle.

How do we know the process is being followed?

The state machine is the process, and scripts move it — Develop is unreachable without approved acceptance criteria, Done is unreachable without a merged PR. The daily digest shows queue health and "your move" items per role; deviations are self-reported in every PR under a mandatory heading.
