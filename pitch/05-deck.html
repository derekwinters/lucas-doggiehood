Why

The three wastes that eat delivery

Translation waste

Turning an idea into work a developer can start: requirements churn, follow-up questions, re-litigated scope.

Readiness drift

"Ready" means something different on every team and every story, so sprints absorb work that isn't buildable.

Waiting waste

Work waits on decisions, decisions wait on meetings, and finished analysis waits for someone free to build it.

The expensive part of delivery lives between the people. That is where this pipeline operates.

Ship

What comes back: one reviewable PR

  • Opens with a mandatory Deviations & Decisions section. The agent self-reports anything off-plan, even when the answer is "nothing".
  • Lists built, dropped, and deferred work, so a partial run never reads as complete.
  • An engineer reviews it against the acceptance criteria that were approved up front, and merges by hand.
  • Your CI, branch protection, and release tooling stay unchanged.

Trust

Safe to run unattended, by construction

  • Authorization. Only allowlisted roles can move work, and ignored commands are reported so attempts stay visible.
  • Idempotency. Processed comments are watermarked. Re-running a routine with nothing new changes nothing.
  • No invented requirements. When the specs run out of answers, the AI stops and asks.
  • No self-merge. Branch protection makes this a platform guarantee.
  • Tested guardrails. The parser, queue, and dashboard are plain scripts with unit tests in CI.
