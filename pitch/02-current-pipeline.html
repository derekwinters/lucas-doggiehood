How an idea becomes shipped code in Doggiehood

This page is the as-built record of the AI issue-management pipeline running today in derekwinters/lucas-doggiehood. Every box, label, command, schedule, and file name below is taken directly from the repository — sources are listed at the bottom so any claim can be checked in minutes.

1 · The state machine, end to end

Three ideas carry the whole design. Labels are the state machine — an issue's label is its pipeline state, so the entire system is inspectable from the issue list. Comments are the control surface — the owner steers everything by typing short /commands on issues, from a phone, at any time. A gatekeeper translates comments into label moves — it runs first in every scheduled routine, honors only the repo owner (the "bad-actor gate"), and watermarks each processed comment with a 👀 reaction so re-runs never double-apply a command.

Large boxes below are the gating sections an issue sits in; small boxes are what is true of the issue while it sits there. Pills on the connectors are the gates that move it forward.

The two loops on the approval gate are the quality mechanism: /revise <notes> sends the issue back to analysis with feedback the next pass must address, /redo discards and re-analyzes fresh, and /propose is the single opt-in that authorizes the AI to draft a missing design — always as a clearly-marked PROPOSAL, never silently.

2 · Which skills and agents run at which phase

Everything AI-facing is packaged as a version-controlled skill (.claude/skills/*) or agent (.claude/agents/*) in the repo itself — reviewed, diffed, and tested like any other code. The deliberate split: wherever a step can be deterministic, it is a plain Python script with unit tests run in CI; the model is spent only where judgment is required (analysis, implementation, short acknowledgments).

3 · The daily clock

Three scheduled Claude Code routines drive the model stages; a plain GitHub Actions cron re-renders the dashboard about an hour after each so it always reflects fresh labels. The gatekeeper runs first inside every routine, so downstream stages never act on stale state. Derek's entire day-to-day involvement is reading the dashboard and typing commands — typically minutes, on a phone.

4 · The owner's command vocabulary

Nine commands are the complete control surface. They can appear anywhere in a comment, surrounded by prose. Parsing is done by parse_commands.py, not by the model — the command grammar, owner-only rule, watermark rule, and parked-issue rule are all enforced in unit-tested code.

5 · Inside the build: the development agent

The doggiehood-dev agent implements exactly one issue at a time, treating the approved Build checklist as its acceptance criteria and the /docs site as the design contract. Test-driven development is not a preference — the agent must show the failing output before implementation exists, and the pipeline drops any issue whose branch can't be kept green.

Red–green–refactor · enforced per behavior change

1failing test first→ 2run · show RED→ 3minimum code→ 4run · show GREEN→ 5refactor · re-run
logic → Doggiehood.Core · engine-free C# Core tests: plain NUnit · dotnet test thin Unity layer → EditMode tests in CI Conventional Commit · Refs #NN docs conflict → stop + flag UI structure w/o wireframe → stop + flag

Two structural guardrails do a lot of quiet work here. The Core/Unity split keeps almost all game logic in an engine-free assembly, so the agent's red-green loop runs anywhere in seconds with no Unity install. And the wireframe-before-UI gate means no screen structure is ever invented during implementation — layout is proposed, reviewed, and approved as a spec + HTML mockup pair in docs/specs/ui/ before code is allowed.

6 · Trust: why the owner can sleep while it runs

  • Authorization — only the repo owner's commands act; anyone else's /commands are parsed, ignored, and reported.
  • Idempotency — the 👀 reaction watermark means a re-run with no new comments changes nothing.
  • No silent design — analysis must stop and ask on any un-specced decision; drafting requires an explicit /propose.
  • No self-merge — the pipeline opens PRs and keeps them green; only a human merges, so nothing reaches main unreviewed.
  • Honest reporting — every PR opens with ## Deviations and Decisions; the nightly cap and dropped issues are logged so a truncated queue never reads as "done".
  • Tested guardrails — the parser, queue selector, and dashboard renderer are pure Python with unit tests (including a golden snapshot) run by CI on every change.
